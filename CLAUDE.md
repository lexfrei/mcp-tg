# CLAUDE.md ŌĆö Development Guide for mcp-tg

## What is this

MCP server for Telegram Client API (MTProto, not Bot API). Uses gotd/td for protocol, exposes 78 tools + resources + prompts via MCP.

## Build & Test

```bash
go build ./cmd/mcp-tg
go test -race ./...
golangci-lint run
```

## Documentation

User-facing documentation lives in `docs/`, is built with MkDocs Material (`mkdocs.yml`) from the fully locked dependencies in `requirements-docs.txt` (compiled from `requirements-docs.in`), and follows the same shape as this owner's other docs sites: sections with their own `index.md` (`navigation.indexes`), a stylesheet, and a `docs.yaml` / `docs-validate.yaml` pair. `.github/workflows/docs.yaml` publishes to <https://mcp-tg.lexfrei.dev> on every push to master; `.github/workflows/docs-validate.yaml` runs `markdownlint-cli2` and the same `mkdocs build --strict` on every PR (`TestDocsGates_RunOnPullRequests`). README.md is a landing page only ŌĆö install, quickstart, links ŌĆö and anything longer belongs on a docs page.

**That PR check must be in `master`'s required status checks, and is** ŌĆö today they are `Lint`, `Test`, `Build (amd64)`, `Build (arm64)`, `Validate Documentation` and `Release Config`. Keep it there: nothing in the workflow graph depends on `Validate Documentation`, so were it dropped, a PR that breaks the site would merge green, `docs.yaml` would fail on master, and the site would quietly stop updating with the last good build still served. The list lives in the repository's branch-protection settings, outside version control like the Pages config below, so it can drift from this paragraph with nothing red; `gh api repos/lexfrei/mcp-tg/branches/master/protection --jq '.required_status_checks.contexts'` is the source of truth. Relative links and anchors inside `docs/` (`[Peers](peers.md)`, `../building.md#transport-modes`) are checked by that build and by nothing else ŌĆö `TestDocsSiteURLs_ResolveToPages` reads absolute URLs only. This is why `docs-validate.yaml` carries **no `paths:` filter**, unlike the reference it is modelled on: GitHub leaves a skipped workflow's check `Pending` forever and blocks a merge on a required check that never reports, so a filtered workflow can never be required at all.

This site is **not versioned**: no `mike`, so no `/latest/` prefix, no per-version tree, and no `404.html` redirect (Material's themed 404 serves from the site root ŌĆö and `mkdocs build` overwrites `docs/404.html` with it anyway, so a hand-written one could not ship). `deploy-pages` waits on GitHub's own Pages deployment and fails if it stalls, so the reference's verify-poll job would only re-check what the action already guarantees.

The custom domain lives in the repository's **Pages settings** (`build_type: workflow`, `cname`, `Enforce HTTPS`) ŌĆö the one piece of site configuration outside version control. `docs/CNAME` ships too, for parity with the house pattern and so the domain is recorded in git if the publishing source ever moves back to a branch, but it is **not what makes the domain work**: GitHub ignores it entirely for a site published from a custom Actions workflow ("no `CNAME` file is created, and any existing `CNAME` file is ignored and is not required"). Do not reach for that file when the domain misbehaves; reach for the settings.

## Key Patterns

### Adding a new tool

1. Create `internal/tools/xxx.go` with:
   - `XxxParams` struct (jsonschema tags: `jsonschema:"Description"`)
   - `XxxResult` struct
   - `NewXxxHandler(client telegram.Client) mcp.ToolHandlerFor[XxxParams, XxxResult]`
   - `XxxTool() *mcp.Tool` with appropriate `Annotations`
2. Create `internal/tools/xxx_test.go` (TDD: tests first)
3. Register in `cmd/mcp-tg/main.go` `registerTools()` via `tools.AddTool(server, registry, ...)` ŌĆö NOT `mcp.AddTool`. The wrapper reflects over the Params struct, records every `bool`/`*bool` field into the registry consumed by the bool-coercion middleware, and then delegates to the SDK. Bypassing it silently disables coercion for the new tool's bool params.
4. Add mock method in `internal/tools/mock_test.go`
5. Add noop method in `internal/testutil/noop_client.go`

### Tool annotations

- `readOnlyAnnotations()` ŌĆö tools that only read data (31 tools)
- `idempotentAnnotations()` ŌĆö tools that modify state but are safe to retry (28 tools)
- `writeAnnotations()` ŌĆö tools that create new entities, not idempotent (10 tools)
- `destructiveAnnotations()` ŌĆö tools that delete/remove things (9 tools)

The four counts must sum to the tool total. `TestToolCensus_MatchesTheDocumentedCounts` (`cmd/mcp-tg/tool_census_test.go`) pins them against the registered server, so a stale number fails CI instead of surviving into the next PR ŌĆö which is how `readOnly` sat one short for a whole release. `TestDocsAnnotationTable_MatchesTheCensus` then pins the published table in `docs/tools.md` against those same constants, so the site cannot drift from the server either.

### Peer resolution

All tools accept `peer` as string. Supported formats:

- `@username`
- `username` (bare)
- `https://t.me/username`
- `https://t.me/+invite_hash` (if already joined)
- Numeric ID (bot-API style: positive=user, negative=chat, -100xxx=channel)

Peers resolved by username get cached with valid access hashes. Numeric IDs reuse cached hashes when available. Prefer @username.

Cold numeric miss (`ResolvePeer` / `wrapper.go`): a numeric ID with no cached hash first tries `resolveViaDialogs` (a single `messages.getPeerDialogs` ŌĆö useless for a channel, whose input peer already needs the missing hash), then `warmDialogsCache`. The warm (`paginateWarmDialogs` ŌåÆ `warmFolder`) paginates the FULL dialog list, both the main folder and the archive (`folder_id=1`), up to `warmDialogsMaxPages` per folder as a safety bound; folder 0 is left implicit so the main request stays byte-identical to a plain listing. gotd omits `folder_id` on the wire when it is 0, so the archive pass MUST set `folder_id=1` explicitly. The whole warm is throttled to once per `warmDialogsThrottle`, and a `singleflight` (`warmSF`, same pattern as `serverConfig`'s `cfgSF`) collapses racing callers onto ONE warm ŌĆö a caller that arrives mid-warm WAITS for it and then sees the populated cache rather than a premature miss. The completion stamp (`warmedAt`) is written only AFTER `warmFolder` scans a folder to EXHAUSTION (a complete `*tg.MessagesDialogs` or a page with no cursor). Every other exit is an error, because the terminal "not cached" verdict may only be drawn from a fully-scanned folder: a page fetch that fails propagates, AND hitting `warmDialogsMaxPages` with the folder still not exhausted returns `ErrDialogScanLimit` (an account with more dialogs than the resolver scans ŌĆö the channel may sit beyond the cap). Neither latches the throttle. `paginateWarmDialogs` scans BOTH folders every time even if one fails ŌĆö an archived channel lives only in folder 1, so a main-folder failure or cap must not skip the archive ŌĆö and remembers the first incompleteness, returning it only after every folder had its chance to seed the cache. Peers cached by pages that did succeed still help: `ResolvePeer` consults the cache before treating a miss as terminal, so a partial warm never loses ground. `warmDialogsCache` returns `(ran, err)` ŌĆö `ran` is false only when the outer throttle skipped the scan (a singleflight-shared scan still counts as ran) ŌĆö and `ResolvePeer` acts on both, only for a `PeerChannel` (`channelResolveError`): the terminal `ErrChannelNotCached` (the "read the listing once, then the id works" contract, like stickers / send-as) is drawn ONLY from a warm that RAN this call AND completed. An INCOMPLETE warm (page error or `ErrDialogScanLimit`) surfaces that error, and a warm SKIPPED by the throttle rests on a scan up to `warmDialogsThrottle` old ŌĆö so a miss there returns the retryable `ErrChannelWarmStale` and invalidates the stamp (`warmedAt = 0`) so an immediate retry re-scans, catching e.g. a channel joined on another session since the last warm. The caller is thus never told a possibly-present channel is absent. Users and basic groups keep the hash-0 fallback regardless of the warm outcome, so the tools-layer `from` / `offsetPeer` guards still label those per parameter.

### Channels vs Groups

Both are handled transparently. Wrapper checks `peer.Type == PeerChannel` and uses the appropriate API methods (e.g., `ChannelsGetMessages` vs `MessagesGetMessages`). No special flags needed except `isChannel` in `chats_create`.

### Auth flow

Two entry points. The server auth uses a cascade: env var ŌåÆ MCP elicitation ŌåÆ error (no stdin fallback ŌĆö stdin is the MCP protocol). The `mcp-tg login` subcommand (`login.go`) is a separate interactive TTY login (phone/code/2FA read from the terminal, no MCP surface) ŌĆö the only way to log in a headless daemon, which cannot elicit. On a headless startup with no valid session the auth error is rewritten (`headlessLoginRequired`) to point at `mcp-tg login` instead of gotd's misleading "TELEGRAM_PHONE is required".

Auth guard middleware blocks tool/resource/prompt calls until auth completes. After auth, a revoked session (`AUTH_KEY_UNREGISTERED` and friends, detected by `auth_revoked.go`) trips `SessionHealth`, and `NewSessionGuard` fast-fails tool calls with `ErrSessionRevoked` (explicit: logged out, run `mcp-tg login`, not fixable from the MCP client).

### Session storage (`storage.go`)

Secure by default: the session lives in the OS keychain via `github.com/lexfrei/keychain` behind gotd's `session.Storage`. macOS uses `WithSecurityCLI` (stable `apple-tool` partition ŌåÆ rebuild-stable for an unsigned daemon). Plaintext file only on explicit opt-in: `--insecure-storage` (login) or `TELEGRAM_SESSION_INSECURE=true` (server); both must match. An unreachable keychain (container / headless Linux) errors with guidance instead of silently writing plaintext. `TELEGRAM_SESSION_FILE` is the keychain account key in secure mode, the file path in insecure mode.

### Client identity

`mcpDevice()` sets `telegram.Options.Device` so the account's Devices list names the client `mcp-tg` (not gotd's default "go1.26.4" device model). The same `DeviceConfig` is passed to `newConnReinitMiddleware` so a connection re-init advertises identical parameters. App-name (from api_id) and Telegram's platform icon are not ours to set.

### Transport modes

`startServer` (`cmd/mcp-tg/main.go`) dispatches on `cfg.HTTPOnly` (`MCP_HTTP_ONLY`):

- Default (`startStdio`): stdio is the primary transport, one process per client, plus an optional additional HTTP transport when `MCP_HTTP_PORT` is set. Auth elicitation runs through the stdio session, so this mode can do an interactive login.
- Headless (`startHeadless`, requires `MCP_HTTP_ONLY=true` + `MCP_HTTP_PORT`): HTTP is the only transport, no stdio peer. One process and one Telegram connection serve many clients ŌĆö the shared-daemon mode. Auth cannot elicit (no client session to prompt), so it depends on a valid persisted session file or env-var credentials.

Both paths build the server through `buildServer`. The headless path passes `onInit=nil` deliberately: the stdio path's `InitializedHandler` closes a shared channel, which would panic on the second HTTP client's `initialize` (close of a closed channel) since headless HTTP serves many clients.

### Resource subscriptions

`tg://chat/{peer}/messages` is subscribable: a new message in a watched chat pushes a `resources/updated` notification for the exact URI the client subscribed with. `SubscriptionBroker` (`internal/telegram/subscriptions.go`) is the app-side registry, modeled on `TranscriptionBroker` ŌĆö process-global shared state, since the headless daemon serves many sessions from one update goroutine. Bookkeeping mirrors the SDK exactly, because the SDK gates delivery by the LITERAL URI a client subscribed with (NOT by peer) and tracks a SET of sessions per URI. The broker mirrors both: `subscribers[uri]` is the set of subscribing sessions, `owner[uri]` records which peer currently triggers each URI, and `peerURIs[key]` is the peerŌåÆURIs reverse index for message lookup. Both keying choices are load-bearing and were reached by fixing the failures of the simpler shapes. **Keying by URI, not peer:** a peer-keyed count breaks because several URI spellings (`@username` vs numeric) resolve to one chat and a URI's `@username` can be reassigned to a different chat; so a message notifies EVERY URI its peer triggers, and `Watch` moves a URI's trigger to the newest peer (carrying the subscriber set across the move). **Keying subscribers by SESSION, not a count:** the SDK calls Subscribe/Unsubscribe handlers BEFORE consulting its own registry, so a duplicate subscribe or an unmatched/cross-session unsubscribe reaches the broker; a plain count would let one client's spurious unsubscribe drop another client's watch. A session set makes `Watch`/`Unwatch` idempotent per session (an unmatched unwatch is a no-op). Session identity is opaque (`any`; the daemon passes `*mcp.ServerSession`) because `ServerSession.ID()` is empty on stdio. `Unwatch` takes `(session, uri)` and resolves no peer ŌĆö both to skip a round-trip and because a resolution failure must never block the SDK from dropping the subscription. `dispatch` snapshots the URIs + notifier under the lock, then calls `ResourceUpdated` per URI OUTSIDE it (first push error returned after all are attempted). Releasing the lock protects concurrent `Watch`/`Unwatch`, NOT the caller's goroutine: `dispatch` runs on gotd's single update-read loop, and `Server.ResourceUpdated` does a synchronous per-session transport write (10s SDK timeout each), so a slow client would stall updates for every client and the co-registered transcription handler. The notifier must therefore be non-blocking: `resourceUpdater` (`cmd/mcp-tg/subscriptions.go`) hands each push to a goroutine under a detached context (the read-loop ctx is cancelled on Telegram reconnect; MCP delivery is independent). This mirrors `TranscriptionBroker`'s non-blocking-send discipline. The `ResourceUpdater` interface is local, so `internal/telegram` carries no MCP import; `main.go` wires the `resourceUpdater` adapter (via `newResourceUpdater`) after `mcp.NewServer`.

Wiring (`main.go`): `dispatcher.OnNewMessage`/`OnNewChannelMessage` are registered BEFORE `tgClient.Run` (same as `OnTranscribedAudio`); the broker threads through `startServer ŌåÆ startStdio/startHeadless ŌåÆ buildServer ŌåÆ newServerOptions`; `SetNotifier` runs after the server is built. `newServerOptions` installs both a `SubscribeHandler` and `UnsubscribeHandler` (the SDK panics if only one is set, and auto-advertises `Resources.Subscribe` once both exist). Each resolves the URI's peer via `resources.ChatMessagesPeer` (which REQUIRES the `/messages` suffix, so the bare info URI is ignored ŌåÆ `nil`) then `client.ResolvePeer`; a resolution error fails the subscribe.

The peer match comes from `msg.AsNotEmpty().GetPeerID()` ŌĆö `tg.MessageClass` has no `GetPeerID`; only `NotEmptyMessage` does. A `MessageEmpty` or a message with no `PeerID` yields the ID-0 sentinel and is dropped. That is exactly the self-send echo shape: `updateShortSentMessage` upconverts to an `UpdateNewMessage` with NO `PeerID`, so the account's OWN outgoing message (sent through this server) never fires the notification ŌĆö a Saved-Messages message from ANOTHER device arrives with `PeerID=PeerUser{selfID}` and does. Live smoke must therefore send from a second device, and the smoke URI uses the numeric self-ID (`resolve.go` has no me/self shorthand).

Known limitation (disconnect drift, cleared only by process restart): the SDK removes a disconnected session from its own subscription registry (`server.go` `disconnect`) but does NOT call `UnsubscribeHandler` (that fires only on the explicit `unsubscribe` RPC). The broker's session set is not pruned, so a dead session's entry lingers. Delivery stays correct ŌĆö the SDK's registry is authoritative, a dead session receives nothing ŌĆö but two costs accrue and neither is bounded by the set of distinct chats: the retained `(URI, session)` entries grow with cumulative abnormal-disconnect churn over the daemon's lifetime (each reconnect mints a new `*ServerSession`); and a lingering entry keeps `subscribers[uri]` non-empty, so that URI's `owner`/`peerURIs` are never cleaned and every later message in the chat costs a peer lookup plus one `subscriber_count=0` INFO line. No clean fix exists ŌĆö `ServerOptions` has no session-closed callback, the headless path never holds the per-session handle, and a prune-on-zero-live-subscribers scheme would race `Watch` (which runs before the SDK registers the subscription). Accept and document; a restart clears it.

### Logging and log level

`resolveLogLevel(os.Args, MCP_LOG_LEVEL)` (`flags.go`) picks the slog level once at startup: `--log-level` flag > `MCP_LOG_LEVEL` env > default `info`; an unknown name is `ErrInvalidLogLevel`, not a silent fallback. `newLogger(level)` builds ONE `*slog.Logger` that is threaded explicitly through `run ŌåÆ startServer ŌåÆ startStdio/startHeadless ŌåÆ buildServer ŌåÆ newServerOptions` (and `waitForTransports`/`runHTTPServer`). It is deliberately NOT a package global. The bug this replaced: `newServerOptions` called `newLogger()` a second time, so the request-logging middleware ignored the level and always logged at info ŌĆö pinned now by `TestNewServerOptions_UsesThreadedLogger`.

Level routing: FLOOD_WAIT retries ŌåÆ WARN (`flood_wait.go`), context cancellation during a FLOOD_WAIT backoff ŌåÆ WARN, HTTP transport shutdown failures ŌåÆ ERROR (`runHTTPServer`), tool-call and method failures ŌåÆ ERROR (`middleware.NewLogging`). The per-request success summary (`MCP request handled`) stays at INFO as an operational heartbeat; DEBUG is the verbose tier (the gotd zap connection/migration logger, `newGotdLogger`). So a default-`info` daemon already records FLOOD_WAIT and failures for a post-mortem without `--log-level debug`.

`--version` (`versionRequested`/`runVersion` in `flags.go`) prints `mcp-tg <version> (<shortRev>)` to stdout and exits before any server or Telegram work; `logStartupVersion` emits the same build id as one INFO line at startup, so an `info`/`debug` daemon names the running binary in its log (at `warn`/`error` it is filtered like any INFO record ŌĆö `--version` and `tg_server_version` are the level-independent build check).

### History pagination

`GetHistory`/`GetTopicMessages` share one `pageHistory` loop (`wrapper.go`): it calls a per-page `historyPageFunc` returning a `historyPageData` (converted messages + `RawCount` + `NextOffset` + `Total`) in chunks of at most `maxHistoryPage` (100, Telegram's server page size), advances the cursor by the RAW page's smallest ID (`minRawMessageID` over `rawHistoryMessages` ŌĆö service messages included), and merges until the limit is reached, the server returns a short RAW page, the cursor is exhausted, `HistoryOpts.MinDate` (a client-side floor ŌĆö getHistory has no server min_date) is crossed, or `HistoryOpts.SinglePage` limits it to exactly one page. `OffsetDate` anchors the first page only; `total` comes from the first page. FLOOD_WAIT is handled per RPC by the existing gotd middleware, so the loop needs no backoff of its own.

The short-page and empty checks are deliberately on `RawCount`, NOT the visible (post-conversion) message count: `convertMessages` drops `tg.MessageService`, so a full 100-item raw page whose visible entries are all service messages would otherwise read as "history ended" and stop the walk while older normal messages still exist. Advancing the cursor by the raw min ID (not the visible min) likewise keeps it moving past a page that is entirely service messages. Two independent caps keep a single call bounded: `pagedHistoryLimit` hard-caps the effective limit at `maxPagedHistory` (1000), bounding the pre-allocation and the merged result; and `maxHistoryScan` (5000) bounds the RAW messages scanned. The scan cap is what actually bounds the round-trips ŌĆö because `merged` grows only by VISIBLE messages, a service-message-heavy stretch would let the loop keep fetching full pages without `merged` reaching the limit, so a merged/limit cap alone would NOT bound the RPCs. This mirrors the type-filter path's `maxTypeFilterScan`. `tg_messages_list` rejects a limit above 1000 outright; any other caller that passes a large limit is silently capped (a fixed-window caller like `tg_messages_context` opts out of paging entirely via `SinglePage`).

Because `merged` grows only by VISIBLE messages, the scan cap can stop the walk with the limit unfilled while older history remains ŌĆö so `hasMore` computed as `len(msgs) >= limit` would falsely read "no more" and strand a caller that pages on it. `GetHistory`/`GetTopicMessages` therefore return a fourth value: `pageHistory`'s `historyMayHaveMore` (last RAW page was full, cursor still live, no floor crossed), which stays true across a scan-cap stop. The non-type `tg_messages_list` path surfaces that bool directly as `HasMore` instead of a len-vs-limit compare (the type-filter path keeps its own `applyScanLimit` hasMore; `messages_context` / resources / prompts ignore the bool). This is why the interface returns `([]Message, int, bool, error)` ŌĆö the mock keeps its 3-value `getHistoryFn` and rides the bool alongside via a `historyHasMore` field.

`tg_messages_list` caps `limit` at 1000 (`maxMessagesListLimit`, `ErrLimitTooLarge`) ŌĆö the only tool that raises the >100 auto-pagination ŌĆö because a bigger limit means more round-trips. The `type` filter path keeps its own outer scan loop; each of its calls bounds `GetHistory` to `telegram.DefaultLimit` (100 VISIBLE messages, not one server page ŌĆö the inner pager may issue several server RPCs to fill that in a service-heavy chat, bounded by `maxHistoryScan`) so a large limit does not page far beyond one outer step. Do NOT set `SinglePage` on that path to make it literally one server page: it would break the outer loop's `hasMorePage(visible, 100)` signal once service messages thin a page. `tg_messages_context` sets `HistoryOpts.SinglePage`: its window is symmetric by construction (`OffsetID = messageId + radius`, `Limit = 2*radius+1`), and auto-paginating to backfill service-message gaps would extend the window only on the older side and turn a large radius into many round-trips ŌĆö so it stays one RPC (server-capped to 100), exactly as before auto-pagination. Resource / prompt reads (`Limit` 50 / Ōēż100) do NOT set `SinglePage`: they want "the last N messages", so filling to N visible past service messages is the intended behavior ŌĆö normally one RPC, occasionally a second bounded page in a service-heavy chat.

`tg_messages_list`'s `toDate`ŌåÆ`HistoryOpts.OffsetDate` and `fromDate`ŌåÆ`HistoryOpts.MinDate` (reusing `validateDateRange`, so `fromDate > toDate` and negatives are rejected before any RPC). getHistory has no server-side `min_date`, so `fromDate` is a client-side floor ŌĆö the read still returns at most `limit` messages, so a wide date window needs a matching `limit`. The default limit stays 100. In the `type` path the toDate `offset_date` is threaded onto the first page only while the fromDate floor rides every page.

### Search filters and global pagination

`tg_messages_search` / `tg_messages_search_global` take `filter` ŌĆö Telegram's server-side `InputMessagesFilter*` kind filter (`internal/telegram/search_filter.go`, a factory map keyed by snake_case names; a map, not a switch, to keep gocyclo flat). This is NOT `tg_messages_list`'s `type`: `type` is a client-side label match over fetched history (`convert.go` labels), `filter` filters on the server; the value sets overlap but don't coincide (`url`, `pinned`, `my_mentions` exist only as filters; `webpage`, `invoice`, `unsupported` only as types). Telegram's `chat_photos` and `phone_calls`/`missed_calls` filters are deliberately NOT offered: they match service messages, which `convertMessages` drops, so every page would come back empty while `total` reports matches ŌĆö do not re-add them without first surfacing `tg.MessageService` in the conversion. The error text for an unknown filter is built from `telegram.SearchFilters()`, so it cannot drift from the factory map. A search needs at least one of `query`/`filter`/`from` (per-chat) or `query`/`filter` (global); a bare `from` is the official clients' "all messages from this member" search.

Per-chat search scoping to a forum topic uses `messages.search`'s native `top_msg_id` ŌĆö no extra `GetReplies` RPC like `tg_messages_list`'s `topicId` path needs. The `from` sender filter and the `offsetPeer` cursor resolve through `resolveOptionalPeer` (`tools/helpers.go`), and each call site then rejects a hash-0 resolution with its own parameter-named sentinel (`ErrFromUnresolved`, `ErrOffsetPeerUnresolved`) ŌĆö `resolveByID` returns hash 0 with a nil error for any numeric ID the client has never seen, and the server error it would cause names neither the parameter nor the fix. The `offsetPeer` guard exempts `PeerChat` (legacy basic groups carry no hash by design and are valid cursor peers); the `from` guard has no exemption, senders are never basic groups.

Global search pagination is a compound cursor (`offsetRate` + `offsetId` + `offsetPeer`); the result returns it ready-made as `nextRate`/`nextOffsetId`/`nextOffsetPeer` (the JSON `messages[].peerId` is a structured object, not the bot-style string `offsetPeer` expects, so callers must not have to convert). When the reply carries no `next_rate`, the wrapper falls back to the last message's date per the documented cursor contract. `SearchGlobal` seeds the peer cache from the reply's `Users[]`/`Chats[]` (same pattern as `tg_chats_get_send_as`) ŌĆö that is what makes a result's numeric `peerId` usable as `offsetPeer` for chats the account never resolved. Do not remove the seeding: without it, page-2 requests fail on unresolvable numeric peers. The cache is in-memory, so a restart between pages loses it ŌĆö a cursor peer that resolves without an access hash is rejected with `ErrOffsetPeerUnresolved` ("re-run the first page") instead of being sent on to a server error that names neither the parameter nor the fix.

### Reply metadata

Reading tools (`messages_list`, `messages_context`, `messages_get`, `messages_search`, `messages_search_global`) expose reply-header fields on each `MessageItem`:

- Structured `replyTo` object (`messageId`, `topId`, `quoteText`, `fromPeerId`, plus advisory `fromName`/`fromUsername`) when the message replies to another. Omitted otherwise. `fromPeerId` is NOT a cross-chat flag: `fillReplyToRef` (`wrapper_helpers.go`) fills the advisory pair whenever Telegram supplies a reply peer at all. **Verified against a live account (2026-07)**: a quote-reply inside a public forum supergroup came back with `fromPeerId` EQUAL to the chat being read, and rendered as `reply to: <id> in <that same chat>`; every other reply in the same batch carried `replyTo` with no `fromPeerId` at all, so "sometimes" is exact and a `fromPeerId != nil` test would be wrong. The one case Telegram documents is narrower than what the wire does ŌĆö `messageReplyHeader.reply_to_peer_id` is described only as "for replies sent in channel discussion threads of which the current user is not a member, the discussion group ID" ŌĆö so the mechanism behind the observed in-chat case is NOT established; do not write one down without evidence. Cross-chat is the `isCrossChatReply` comparison ŌĆö `fromPeerId` differing from the read peer in type or ID ŌĆö which is why `resolveReplyParents` re-derives it per message rather than testing `fromPeerId != nil`.
- Text `output` emits a `reply to: <parentId>` line (or `reply to: <parentId> in <peer-ref>` whenever `fromPeerId` is set ŌĆö `writeReplyLines` branches on that field being non-nil, NOT on the cross-chat comparison, so an in-chat quote-reply renders an `in ...` reference pointing back at the chat being read; pinned by `TestFormatMessage_SameChatQuoteReplyStillRendersReference`), followed by a `quote: ┬½...┬╗` line if `quoteText` is present. This applies to `messages_list`, `messages_context`, `messages_get`, and `messages_search`. `messages_search_global` returns only a summary line (`Found N of M message(s)`, M being the server's total) ŌĆö its `output` does not format individual messages, so callers must read the JSON `replyTo` field for global-search replies. See `docs/guides/messages.md` ("Message Output Format") for the full block layout.

The first four tools also accept an optional `resolveReplies` parameter (default `false`). When `true`, every reply whose parent is reachable is attached as `replyToMessage: { fromName, fromUsername, text }`, with the text truncated to `replyParentTextLimit` runes (`resolve_replies.go` owns that number and `getMessagesMaxIDs`; the docs page states both and `TestDocsReplyNumbers_MatchTheCode` pins them together, so do not restate either here). Enrichment covers parents ALREADY in the returned batch, at no extra cost ŌĆö only the ones missing from it are fetched, in `slices.Chunk` batches of `getMessagesMaxIDs`. That cap is Telegram's, not ours: `messages.getMessages` takes "up to 200 IDs ... re-invoking the method until the desired range is fetched" (`core.telegram.org/api/updates`), and `wrapper.GetMessages` sends whatever list it is handed. The number is DOCUMENTED, not observed ŌĆö the method pages state no limit, and reaching the boundary needs >200 distinct out-of-batch parents in one call, which no tool can be made to produce on demand (the search cap below was verifiable; this one is not). TDLib corroborates it independently and more strongly: it splits BOTH methods at `MAX_SLICE_SIZE = 200`, annotated `// server-side limit`, on its general read path ŌĆö which is what rules out reading TelegramŌĆÖs prose as advice narrow to gap-filling. TelethonŌĆÖs 100 is not a counter-example: that is its getHistory page-size constant reused, claiming nothing about getMessages. The mock states its cap in terms of `getMessagesMaxIDs`, so the tests pin the chunking, not the 200: were the real ceiling lower, they would stay green and the silent failure would return. Treat it as an upper bound to respect ŌĆö chunking smaller is safe, larger is not. It is reachable ŌĆö `tg_messages_list` allows `limit` up to 1000, so one call can collect more than 200 distinct out-of-batch parents, and it is the ONLY tool that can: **verified against a live account (2026-07)**, `messages.search` caps its reply at 100 however large a `limit` it is handed (a search with `limit: 300` over 2831 matching messages returned exactly 100), and nothing in this repo enforces that ŌĆö `validateLimit` rejects only negatives and `Limit` reaches the wire untouched, so the ceiling is Telegram's own and is undocumented on the method page. `tg_messages_context` is bounded by `maxHistoryPage`, `tg_messages_get` by `maxIDsPerRequest`. An over-long request is rejected whole, `fetchMissingParents` swallows the error, and every out-of-batch parent then loses its `replyToMessage` while in-batch ones keep theirs ŌĆö silent and partial, never an error. A failed chunk now costs only its own parents (`continue`, not `return nil`), while a dead context and a FLOOD_WAIT both BREAK the loop. The split is the point: a bad id set is the chunk's own problem, but a throttle is a statement about the CONNECTION ŌĆö every remaining chunk would hit it, and `newFloodWaitMiddleware` sleeps through its retries before passing the raw error up, so a `continue` would burn a server-chosen delay per chunk to collect nothing. `tgerr.AsFloodWait` (already used by `wrapTelegramError`, same package) is what tells them apart. The split is COST, not category: `CHANNEL_INVALID` and friends are equally certain to repeat on every later chunk but fail fast, so they still `continue`; `ctx.Err()` cannot, because FLOOD_WAIT never cancels the context and no caller here sets a deadline. Chunking is what made that multiplication possible, so it carries the fix. Four pins, because each branch is invisible to the others: `TestResolveReplyParents_ChunksAtTelegramsIDCap` (mock rejects an over-long id list ŌĆö the repo mock modelled no cap, which is why nothing caught this; hence `getMessagesErrFn`), `..._FailedChunkKeepsTheOthers` (one chunk fails, the other's parents must survive ŌĆö indistinguishable from total discard with a single chunk, which is why the older best-effort test could not catch it), `..._StopsOnDeadContext` / `..._CancelledMidLoopKeepsWhatItHas` (cancel before the first chunk vs between chunks ŌĆö only the latter can tell "return what we have" from "return nothing"), and `..._StopsOnFloodWait` (three chunks, throttle in the MIDDLE: flooding every chunk collects nothing either way so `break` and `return nil` look alike, and flooding the last makes `break` and `continue` alike). Cross-chat replies (`isCrossChatReply`: `replyTo.fromPeerId` differs from the read peer in type or ID) are skipped since we lack the foreign peer's access hash. `messages_search_global` does not offer `resolveReplies` for the same reason ŌĆö its results span arbitrary peers, a batched lookup is not feasible. The resolver is best-effort: `fetchMissingParents` swallows the error, so `replyToMessage` is simply absent on failure.

`resolveReplies` enriches only the JSON `replyToMessage` field, and `shouldResolveReplies` skips the work entirely under `format: "text"` (which drops the `messages` array it writes into). The text `output` is built once from the fetched batch and reads identically whether the flag is on or off ŌĆö the same `reply to:` line, its `in <peer-ref>` form and any `quote:` line the reply already carried, never the resolved parent's text. Callers that need it should read the JSON structure.

### Forwards and peer identifiers

Reading tools populate `MessageItem.forward` from `tg.MessageFwdHeader` so callers can distinguish a forwarder from the original author:

- `forward.from` ŌĆö resolved `PeerRef` (peer+name+username) of the original sender; nil when the original author has forward-privacy enabled.
- `forward.fromName` ŌĆö display name leaked through when `from` is nil (privacy-hidden forward).
- `forward.date`, `forward.channelPost`, `forward.postAuthor` ŌĆö original timestamp, ID of the source channel post, and the channel's signed-author byline.

**Single helper, single shape, everywhere.** Every peer rendered to the user ŌĆö sender, forward origin, reply target, dialog, group, channel, user, contact, reactor ŌĆö goes through `formatPeerRef(name, username, peer)` and produces the same string: `Display Name [@username]` / `[user:N]` / `[channel:N]` / `[group:N]` / `[hidden]` / `[unknown:N]`. The kind label (`user` / `group` / `channel` / `unknown`) matches the JSON `type` field on `ParticipantItem`, `MessageItem.fromType`, and every other peer-bearing JSON entry. `group:` covers `PeerChat` only; supergroups label as `channel:` (gotd folds them into `PeerChannel`).

`formatUserName(*telegram.User)` and `formatDialog(*telegram.Dialog)` are thin adapters that route their inputs into `formatPeerRef`, so a renamed `peer*` constant in `format.go` propagates to every surface. Resource `tg://chat/{peer}/messages` uses the exported `tools.FormatMessageList` so it shares the same multi-line block layout the read tools emit.

Username resolution for `messages_*` piggybacks on the `Users[]`/`Chats[]` arrays MTProto already returns in `MessagesMessagesClass`, so no extra API round-trip. `tg_dialogs_*` and `tg_groups_info` get the username directly from the wrapper-returned `Dialog`/`GroupInfo`. `ContactStatus` has the field slot but it is currently unpopulated ŌĆö `ContactsGetStatuses` does not return `Users[]` alongside statuses, so consumers see `[user:N]` rather than `Name [@user]` for those entries until a follow-up adds a batched lookup.

### Multi-line text output

Each message in `output` is a block of `key: value` lines (`from:`, `forwarded from:`, `reply to:`, `quote:`, always-present `type:`) followed by the body under `text:`. Blocks are separated by a literal `---` line so a message body containing its own blank lines (Telegram bodies routinely have paragraph breaks) stays unambiguous. Long bodies are emitted verbatim ŌĆö no truncation in the human-readable string. The single-line `[ID Ōå®parent] ts sender: text` form was removed; do NOT reintroduce it or grep for `Ōå®`. Use `MessageItem.type` / `type:` for message kind; do NOT reintroduce `mediaType` / `media:`.

### Output format param

The five read tools (`messages_list`, `messages_get`, `messages_context`, `messages_search`, `messages_search_global`) take an optional `format` enum (`full`/`json`/`text`, default `full`) constrained on the wire via `inputSchemaWithEnum` (the `parseMode` precedent) ŌĆö optional, so it is NOT in the schema `required` set; an omitted value is `full`. `messagesForFormat`/`outputForFormat` (`messages_format.go`) trim the result: `json` clears `Output`, `text` clears `Messages`, `full` keeps both. The two helpers are split (not one two-value function) because gocritic `unnamedResult` and `nonamedreturns` conflict on a multi-value return. Both result fields are now tagged `omitempty`, so a cleared field vanishes from the JSON ŌĆö this is a deliberate shape change (documented in `docs/guides/messages.md`, "Message Output Format"): a full-mode result with zero messages now also omits `messages`. Handlers apply format LAST, after `resolveReplies`, so the enrichment still runs for `json`/`full`.

### Message entities (formatting on read)

`MessageItem` carries an optional `entities` array with the message's formatting spans as read from MTProto `Message.Entities`. Each entry has `type` (Bot API naming: `bold`, `italic`, `code`, `pre`, `text_url`, `url`, `mention`, `hashtag`, etc.), `offset` and `length` in UTF-16 code units, plus optional `url`/`language`/`userId` for types that carry metadata. Plain messages omit the field entirely.

### Parse mode (formatting on write)

Tools that send or edit text (`messages_send`, `messages_edit`, `messages_send_file`, `media_send_album`) REQUIRE `parseMode` ŌĆö no default, deliberately: optional formatting params get systematically omitted by LLM callers and markdown then ships as literal asterisks. The choice is enforced at the protocol layer ŌĆö each tool's input schema carries `enum: [plain, commonmark]` (built by `inputSchemaWithEnum` in `tools/register.go`, which infers the schema exactly as `mcp.AddTool` would and patches one property), validated by the SDK before the handler runs. The schema enum is strict lowercase; `normalizeParseMode`'s case-insensitivity survives only as defense in depth for direct handler calls (tests). `TestParseModeSchema_RequiredEnumOnTheWire` pins required+enum on the wire representation.

- `"plain"` ŌĆö no formatting. Text (or caption) that looks like markdown is REJECTED with `ErrPlainLooksLikeMarkdown` unless `allowRawMarkdown=true`; the lint (`telegram.LooksLikeMarkdown`, `markdown_lint.go`) conservatively matches fences, inline code, doubled-marker emphasis (`**`/`__`/`~~`/`||` with non-whitespace-flanked content), `[text](url)` links and `<https://...>` autolinks, and deliberately excludes single `*italic*`/`_italic_` and `>` quotes as false-positive-prone. `__init__` is a true positive by design ŌĆö the parser really would transform it.
- `"commonmark"` ŌĆö CommonMark subset: `**bold**`, `*italic*`, `` `code` ``, ` ```pre``` `, `~~~pre~~~`, 4-space indented code blocks, `[text](url)`, `<https://autolink>`, `> quote`, `>quote` (no space ok), `~~strike~~`, `__underline__`, `||spoiler||`. Parsed into `tg.MessageEntity` on the server side.
- `"markdown"` ŌĆö RETIRED alias; rejected with a steering error pointing at `commonmark`. The wrapper's `IsCommonMarkParseMode` still recognises it internally as harmless defense.
- `"html"` / `"markdownv2"` ŌĆö recognised but not yet implemented; return a clear error (direct-handler path only ŌĆö over MCP the schema enum rejects them first, like every non-enum value).
- `""` (omitted) ŌĆö rejected by the schema (`required`), and by `ErrParseModeRequired` on the direct-handler path.

All four results carry `entitiesParsed` ŌĆö the count of FORMATTING entities in the SERVER echo, serialized even at 0. Auto-detected types (`url`, `mention`, `hashtag`, `cashtag`, `bot_command`, `email`, `phone`, and anything unmapped ŌĆö `IsFormattingEntity` is an ALLOW-list precisely so a future auto-detected kind cannot silently start counting) are excluded: the server adds them to any message regardless of `parseMode`, and counting them reported `entitiesParsed: 2` for a plain send that merely contained a link and a hashtag ŌĆö verified live. Only the types a parseMode can produce count (0 after a commonmark send means nothing parsed; the caller can self-correct via `tg_messages_edit`). The echo is the source, not a client-side recount: `messageFromUpdate` converts `updateShortSentMessage.Entities`, and `editedMessageFromUpdate` handles `updateEditMessage`/`updateEditChannelMessage` (before that fix `EditMessage` returned nil ŌĆö an edit-based self-correction loop would have seen 0 forever). **Identity in an echo envelope comes from `random_id`, not from position.** Every send generates a `random_id`, and the server pairs it with the ID it assigned in `updateMessageID` ŌĆö that pairing is the ONLY identity signal MTProto gives. An envelope can also bundle other messages entirely: a channel post fans out into the linked discussion group, an album's copies land there too, a topic root's reply counter bumps. Taking the first new message would report a stranger's ID, which the caller would then edit, delete or pin. `ownMessageIDs` filters by our `random_id` set (`messageFromUpdate` for single sends, `messagesFromUpdates` for albums and forwards); when the server sends no `updateMessageID`, the old first-match behavior is the fallback. Every path additionally checks the peer, because message IDs are numbered PER PEER: a channel's counter runs close to its linked discussion group's, so the group's copy of our post can carry the very same ID. Matching on the number alone would read the entity count off a stranger, over-count an album, or (on edit) report the group's copy as the post you edited.

The two extractors are deliberately separate: a send echo may carry an edit update for the topic root (its reply counter bumped), and the send path must never mistake that parent for the message it just sent. `editedMessageFromUpdate` additionally matches on the edited ID rather than taking the first edit update ŌĆö an envelope can bundle edits of other messages (a channel post's edit propagating into the linked discussion group), and the tool keeps the caller's `messageId` rather than deriving it, since Telegram does not renumber an edited message. Albums report the SUM across echoed messages, since the server's update order is not a contract.

**Verified against a live account (2026-07):** a Saved Messages send ŌĆö precisely the case `updateShortSentMessage` exists for ŌĆö answers with the full `*tg.Updates` envelope, in both plain and commonmark mode. The short form never fired, so whether its conditional `entities` flag is set on an unchanged entity set could NOT be observed, and Telegram documents neither. So `shortSentEntities` (`wrapper_helpers.go`) repairs THAT SHAPE ONLY: when the short echo carries no entities, the count falls back to the set the request submitted, which is safe because Telegram rejects malformed entities outright (`ENTITY_BOUND_INVALID` and friends) rather than dropping them silently. The fallback must NOT be widened to the full-updates path: there the server sends back its own message, and a zero means the server really applied nothing ŌĆö the exact signal `entitiesParsed` exists to carry. Widening it would erase the signal in the name of protecting it.

`messagesFromUpdates` (albums, forwards) shares `extractNewMessage` with the single-message path so scheduled sends (`updateNewScheduledMessage`) are not silently dropped ŌĆö a scheduled album used to come back with `count: 0`. Of these two callers only albums can be scheduled ŌĆö `tg_messages_forward` has no `scheduleDate` (`tg_messages_send` and `tg_messages_send_file` do, but they take the single-message path).

On the SINGLE-message paths (`SendMessage`, `EditMessage`, `SendFile`) an echo that cannot be read at all ŌĆö an unhandled envelope shape, or no update matching the edited ID ŌĆö is repaired by `echoOrSubmitted` with the submitted entity set. Without it a nil echo reads as `entitiesParsed: 0`, which the documented recipe treats as "the markdown did not parse", and a caller would re-edit correct text forever with nothing erroring. A readable echo always wins, including one that genuinely reports zero.

`SendAlbum` is deliberately NOT repaired: there is no single ID to attach a repaired count to, and the album's anomaly already surfaces as `count: 0` ("Sent album with 0 file(s)"), which is visibly wrong rather than quietly plausible ŌĆö pinned by `TestMediaSendAlbumHandler_UnreadableEchoReportsNothing`. The single-message repair has its own oddity for symmetry's sake ŌĆö `entitiesParsed: N` beside `messageId: 0` ŌĆö which is equally loud: a follow-up edit on id 0 fails outright rather than silently editing the wrong message.

Known CommonMark gaps documented in `docs/guides/messages.md` ("Markdown ŌĆö Known Limitations"): nested blockquotes (`> > x`), nested emphasis (`**a *b***`), hard line breaks via two trailing spaces or trailing `\`. Each has a commented-out test in `internal/telegram/markdown_audit_test.go`.

### Send-as identity (posting as a channel)

MTProto's `send_as` field appears on exactly five requests, which map onto six tools: `messages.sendMessage`, `messages.sendMedia` (backs both `tg_messages_send_file` and `tg_stickers_send`), `messages.sendMultiMedia`, `messages.forwardMessages`, `messages.createForumTopic`. All six take an optional `sendAs` string.

Wiring: `SendOpts.SendAs *InputPeer` for the three methods that already had an options struct; a trailing `sendAs *InputPeer` argument for `ForwardMessages` / `CreateForumTopic` / `SendSticker`. `applySendAs(sendAs, req.SetSendAs)` (`wrapper_helpers.go`) sets the conditional field for all five request types via the method value ŌĆö do NOT inline the `if sendAs != nil` check per call site, `dupl` will flag it.

`nil` does NOT mean "post as the account". It leaves the flag clear, and the server then applies the chat's saved default ŌĆö verified on a live account: with a channel set as the chat default, a send with no `send_as` posts as that channel. This matches the official clients. Do not "fix" it by substituting `inputPeerSelf`: that would make `tg_chats_set_send_as` govern reactions but not messages, which is neither Telegram's model nor a useful one.

`resolveSendAs` (`tools/helpers.go`) resolves the string and rejects a `PeerChannel` whose `AccessHash` is 0. That state is reachable and silent: `resolveByID` returns it with a nil error for any numeric ID the client has never seen, and the resulting server error names neither the parameter nor the fix. `tg_chats_get_send_as` seeds the peer cache from the `Chats`/`Users` of its own reply, which is what makes numeric IDs work afterwards.

`tg_chats_set_send_as` wraps `messages.saveDefaultSendAs`. Omitting `sendAs` there is not a missing argument ŌĆö it resets the chat to the account (`&tg.InputPeerSelf{}` in the wrapper, since the domain `InputPeer` cannot express self). The default is account-wide server state and the only way to react or vote in a poll as a channel; `messages.sendReaction` and `messages.saveDraft` have no `send_as` field at all. `tg_groups_info` reports the current default from `ChannelFull.DefaultSendAs` at no extra RPC cost.

Both new tools reject non-`PeerChannel` peers before the round trip (`telegram.ErrSendAsUnsupportedPeer`); the server's `CHANNEL_INVALID` explains nothing.

Posting as yourself for a single message, in a chat whose default is a channel, works: pass your own numeric ID as `sendAs`. It resolves with an access hash because the account always has a Saved Messages dialog, so `resolveSendAs` accepts it and `InputPeerUser` is what reaches the wire. Verified end-to-end. `inputPeerSelf` is therefore needed only by `SetDefaultSendAs`, where the identity is absent rather than resolved.

Verified against a live account: a rejected identity comes back as `CHAT_ADMIN_REQUIRED` (channel you don't administrate) or `CHAT_WRITE_FORBIDDEN` (foreign user), NOT `SEND_AS_PEER_INVALID` ŌĆö that code exists in the schema but the server rarely reaches for it. Both read as a chat-permission problem, so `sendErr` (`tools/errors.go`) names `sendAs` as a suspect whenever one was supplied. Do not "simplify" the six send tools back to plain `telegramErr`.

Also verified: a channel that is the chat default reacts as itself, and `messages.getMessageReactionsList` returns its title in `Chats`, not `Users` ŌĆö `extractReactionUsers` reads both and carries `ReactionUser.PeerType`.

### Sticker documents (`sticker_cache.go`)

`inputDocument` needs id + access hash + file reference. Only the id is public and stable; the other two arrive with `messages.getStickerSet` and cannot be derived. `GetStickerSet` therefore seeds `StickerCache`, and `SendSticker` looks the document up there, failing with `ErrStickerNotCached` before the RPC when it is absent. Sending a bare id (which is what the code did until this cache existed) answers `MEDIA_EMPTY` on every call. Same shape as the send-as peer cache: read the listing tool once, then the id works.

**`stickerFileId` is a string, and must stay one.** `mcp/tool.go` in go-sdk v1.6.1 unmarshals tool arguments into `map[string]any`, calls `ApplyDefaults`, then re-marshals ŌĆö so every JSON number passes through `float64`. `internal/json` wraps `segmentio/encoding/json` without `UseNumber`, so there is no escape hatch. A sticker document id needs 63 bits; the mantissa holds 53. Verified end-to-end: `5181593617004757506` reached the wrapper as `5181593617004758000`. Any future tool parameter above 2^53 must be a decimal string for the same reason ŌĆö this is not paranoia, `stickerFileId` was silently broken by it. Telegram user IDs are well under the limit and are safe as numbers.

### Telegram protocol details

- **RandomID**: All send operations (message, file, album, forward, sticker) generate crypto-random IDs for deduplication
- **FLOOD_WAIT**: `newFloodWaitMiddleware(logger)` makes at most `maxFloodRetries` attempts (3 ŌĆö so two retries) with the server-specified delay, logging each retry at WARN with `retryAfter` (and a WARN if the context is cancelled mid-backoff). A FLOOD_WAIT that survives all retries reaches the tools layer, where `wrapTelegramError` marks it `ErrFloodWait` with a readable "retry after Ns" (not a raw code, not JSON) ŌĆö the server stays up, so the caller can back off and retry
- **Peer cache**: Access hashes from username resolution, dialog listing, and `channels.getSendAs` are cached in memory

## Release and distribution

`release.yml` fires on a `v*` tag: it builds and signs the multi-arch container, creates the GitHub release with generated notes, and then a final job runs GoReleaser (`.goreleaser.yaml`) to upload the binary archives (darwin/linux/windows ├Ś amd64/arm64) and publish the Homebrew formula to `lexfrei/homebrew-tap`. `release.mode: keep-existing` ŌĆö the mode governs the release NOTES, not the artifacts, and `append` would stack a second, GoReleaser-generated changelog under the handwritten one on every release.

The two channels must inject the same version strings: the container passes `v{{version}}` and the full SHA (that is what `docker/metadata-action` emits), so GoReleaser's ldflags use `v{{.Version}}` and `{{.FullCommit}}` too. Otherwise one release introduces itself two different ways over the MCP handshake and in Telegram's device list. `TestLdflagsSymbols_ExistInMain` pins the symbol names both files inject against the vars in `main.go` ŌĆö the linker ignores `-X` for a symbol that does not exist WITHOUT a word, so a rename would leave every release reporting `dev`.

`replace_existing_artifacts: true` is what makes the release job re-runnable. Asset upload is the only non-idempotent step in the publish chain (the release edit and the tap push are both upserts), it answers 422 `already_exists`, and GoReleaser treats that as unrecoverable ŌĆö so a re-run after a failed tap push, the obvious recovery, would die on the first archive and never reach the tap it was re-run for.

`pr.yml` runs `goreleaser release --snapshot --clean` as a gate (a snapshot implies `--skip=publish`; signing is skipped only for fork PRs, which cannot mint an OIDC token). Without it the release config would first execute on the tag, in the LAST job ŌĆö after the container is pushed and the release is public ŌĆö so a broken template would leave a published release with no binaries and no formula. `goreleaser check` cannot be the gate: it exits non-zero on the deliberate `brews` deprecation.

The formula ships an env file at `etc/mcp-tg/mcp-tg.env` and a wrapper in `libexec` that sources it, and the service runs the WRAPPER. This is not decoration: a service manager injects only the variables the unit declares, credentials cannot live in a public formula, and `launchctl setenv` dies with the login session ŌĆö so a service running the bare binary comes up unconfigured after every reboot and `keep_alive` restarts it forever. The PR gate also runs `ruby -c` over the generated formula: `install`/`service`/`test`/`caveats` are hand-written Ruby pasted into YAML, GoReleaser substitutes them as strings without parsing, so a missing `end` writes a broken formula and still exits 0 ŌĆö every `brew install` would then fail with a SyntaxError. The formula's `test do` block asserts three things at once (terminates without touching the network, exit code 1, the `TELEGRAM_APP_ID is required` sentence); `TestBrewFormulaContract_NoCredentialsExitsOneWithoutNetwork` pins all three from this repo, because the tap is where a break would surface and nobody reads it.

The tap gets a FORMULA, not a cask, which is the deprecated path in GoReleaser (`goreleaser check` fails on it ŌĆö the release command only warns). The reason is that both replacements cost something the daemon needs: casks are macOS-only, and they have no `brew services`. A formula gives Linux and `brew services start mcp-tg` (launchd on macOS, systemd on Linux), which is how the shared HTTP daemon is meant to run. Homebrew itself still supports formulae; only GoReleaser's generator is retiring them. Windows gets archives but no tap: Homebrew has no native Windows support at all.

`HOMEBREW_TAP_GITHUB_TOKEN` (a fine-grained PAT scoped to the tap, Contents: read and write) must exist as a repository secret, and `preflight` verifies it can actually PUSH, by probing `info/refs?service=git-receive-pack` ŌĆö the first exchange of a real `git push`, which mutates nothing. `GET /repos` will NOT do: its `permissions` field reports the authenticated user's ROLE in the repository, not the token's grants, and a fine-grained PAT can read any public repo regardless of its scope ŌĆö so a read-only token would sail through. A non-empty secret is likewise not the interesting case: a fine-grained PAT expires within a year, and an expired one is still non-empty. A missing secret expands to an empty string, GoReleaser builds a GitHub client with an empty token, and the tap push fails. The snapshot gate cannot catch that ŌĆö it never publishes ŌĆö so a `preflight` job checks the secret before ANY job runs: the container push and the public release both happen after it, and a failure there would otherwise land on top of an already-published release.

A pre-release tag (`v1.3.0-rc1`) must not move `latest` anywhere. FIVE places key off the semver hyphen and must agree: `prerelease:` in `create-release`, `prerelease: auto` in `.goreleaser.yaml`, `enable=` on the `type=raw,value=latest` container tag in EACH of the two metadata steps, and `skip_upload: auto` on `brews` ŌĆö for Homebrew the formula in the tap IS `latest`, so an RC would otherwise land on every `brew upgrade`. `TestPrereleaseGuards_AllFiveSitesAgree` pins all of them: a drift in any one would make an RC the latest image AND the installed formula, with nothing red anywhere.

The release archives are signed too (keyless cosign over the checksums file), because the container is signed and the notes teach `cosign verify` ŌĆö a binary that hands out a full-access Telegram session should not be the unsigned half of the same release. The signature is a `.bundle`: cosign v3 IGNORES `--output-signature`/`--output-certificate` and still exits 0, so the pre-v3 flags would have shipped an unsigned release whose notes told people to verify files that were never written. The PR gate therefore asserts the bundle EXISTS rather than trusting cosign's exit code.
